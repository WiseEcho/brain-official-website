# 项目经验教训

## 2026-06-30：高级 UI 优化不能只调间距，要重构信息层级

**问题**：
用户反馈第一次 Hero 优化「并不是太满意，要从高级 UI 设计视角出发」。我之前只是机械地缩小 padding、margin、字号，没有解决核心视觉问题。

**原因**：
- 只做了「压缩尺寸」的表层处理，没有重构信息层级和阅读动线
- 没有识别真正造成拥挤的原因：独立的大数字 `300%`、带副标题的双 CTA、过长的标题换行
- 缺少对 viewport 适配的系统性思考（Hero 应该在一屏内完整呈现核心信息）

**规则**：
- 做 UI 优化时先分析**视觉重心、信息层级、负空间分布**，再动代码
- Hero 区必须遵循：标题 ≤ 2 行、副文案精简、CTA 可见、整体可在一屏内看完
- 大数字/关键词应融入标题或作为视觉锚点，避免成为孤立元素
- CTA 尽量只保留一个主行动点，次按钮要明显弱化

**如何应用**：
1. 先列出 Hero 当前所有视觉元素，判断每个元素的信息优先级
2. 将大数字、核心卖点整合进主标题，减少独立元素数量
3. 简化 CTA（去掉副标题、统一为单行按钮）
4. 使用 `min-h-[calc(100dvh-4rem)]` 等方式确保首屏完整呈现
5. 改完后在 1280×832、1440×900 等小笔记本分辨率下截图验证

## 2026-06-30：设计系统必须先审计现有风格

**问题**：
用户在看到设计系统演示后反馈「样式和原来差别较大，要严格遵守原来的风格样式」。

**原因**：
我在做设计规范调研和 demo 时，过度依赖通用 SaaS 设计趋势（卡片阴影、圆角、简洁白底），没有先充分审计现有官网页面的真实视觉语言（颜色使用方式、间距、组件形态、动效、图文混排节奏）。

**规则**：
- 做任何设计系统/样式重构前，必须先完整阅读现有页面 HTML 和 CSS，提炼出**原有设计特征清单**
- 新增规范应是对现有风格的系统化、补足，而不是另起一套新风格
- demo 页面必须使用现有真实素材和类名，确保用户能直观对比「和原来一致」

**如何应用**：
1. 读取 index.html、products.html、pricing.html、about.html 等关键页面
2. 记录原站使用的：主色/辅色/背景色、标题字号、卡片样式、按钮样式、间距节奏、动效类型
3. 新设计系统文档中明确标注「继承自现有设计」vs「新增规范」
4. demo 页复用现有 main.css 和 Tailwind 类名，只做结构层面的演示
